## Themes

### 1. mx-chain-go v1.11.7 — interceptor bootstrap stability release

v1.11.7 is not a Supernova transition release; it's a targeted bootstrap and interceptor lifecycle fix. All four bundled PRs address how nodes initialise and tear down interceptors:

- **[#7861](https://github.com/multiversx/mx-chain-go/pull/7861)** "recreate interceptors on bootstrap" by AdoAdoAdo — a node restarting mid-epoch was not recreating its message interceptors, leaving it unable to process certain messages until the next epoch boundary. Critical for restart and recovery reliability.
- **[#7876](https://github.com/multiversx/mx-chain-go/pull/7876)** "Fix transition on msg reverse order" by AdoAdoAdo — edge case where interceptors process messages in reverse order during an epoch transition, first identified in review of `start-in-epoch-reload-interceptors` branch.
- **[#7865](https://github.com/multiversx/mx-chain-go/pull/7865)** "unregister all processors on tear down" by sstanculeanu — ensures clean shutdown by unregistering message processors on graceful node stop.
- **[#7862](https://github.com/multiversx/mx-chain-go/pull/7862)** "more bootstrap optimizations" by raduchis — additional speedups to the bootstrap path (building on earlier sync improvements).

The same-day merge → tag → mainnet-config PR sequence indicates the team treats this as production-ready. The `start-in-epoch-reload-interceptors` branch that originated PRs #7861 and #7876 is now effectively merged through v1.11.7.

### 2. v1.11.7 → mainnet deployment in motion

[mx-chain-mainnet-config #269](https://github.com/multiversx/mx-chain-mainnet-config/pull/269) "Release v1.11.7.0" by sstanculeanu was opened June 10, hours after the tag dropped. Historical pattern: a mainnet-config release PR follows each pre-release tag by 0–2 days; the mandatory-upgrade announcement typically follows 2–5 days after that. The prior release (v1.11.6, mainnet-config PR #268) shipped May 27; that cycle was 14 days from PR open to deploy. Watch for a validator upgrade notice this week.

### 3. Proposal integrity — new shard data validation PR

[mx-chain-go #7877](https://github.com/multiversx/mx-chain-go/pull/7877) "extra checks shard data proposal" by miiu96 opened June 10 is separate from the Supernova cluster — it targets `master` and adds validation of shard data inside block proposals. This is consensus safety hardening: ensuring proposal data is well-formed before peers vote on it. Likely a finding from the Supernova testnet hardening sprint applied back to the production path.
