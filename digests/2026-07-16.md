## Themes

### A real storage-integrity fix lands in `mx-sdk-rs`

[#2413](https://github.com/multiversx/mx-sdk-rs/pull/2413) fixes two related bugs in `MapStorageMapper`: `insert_default` previously inserted a key into `keys_set` without checking whether stale data already existed at that key's nested storage namespace — meaning a newly "inserted" entry could silently inherit leftover state from a prior occupant (e.g. after an out-of-band write or a previous remove that left a detached handle live). The fix makes `insert_default` clear any residual nested storage before registering the key, and adds documentation warnings on `OccupiedEntry::get`/`remove` about the detached-handle hazard. Three new regression tests cover it, including one that injects a raw stale key to simulate the exact failure mode. This is the kind of fix that matters more than its diff size (+129/−5 across 2 files) suggests — any contract using `MapStorageMapper` with delete-then-reinsert patterns was exposed.

### `mx-chain-go`'s consensus stack keeps growing, still nothing merged

Five days into this storyline (first flagged Jul 14), the stack is now five PRs deep — `cross-shard-gate` → `contended-blocks-finality` → `rollback-v3` → `meta-arbitration` → today's new `chain-simulator-contended-settlement-test` — and #7919 itself gained two more commits without merging. This is consistent, deliberate incremental review rather than a stall (each addition is small and reviewable), but the "zero merges to master" metric continues to understate the actual velocity here. The two near-duplicate go.mod PRs (#7921/#7922) are a minor housekeeping wrinkle worth a follow-up check.
