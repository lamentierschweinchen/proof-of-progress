## Themes

### 1. rc/v0.67 cryptographic rewrite completes in one day

Two PRs merged into rc/v0.67 on May 28 finish the signing-layer revamp:

- [#2382](https://github.com/multiversx/mx-sdk-rs/pull/2382) **Signing using ed25519-dalek** — replaces the SDK's custom `edwards25519` implementation with the widely-audited `ed25519-dalek` crate (Ristretto group). Adds `PublicKey::verify` API and expands signing tests to cover signature verification. The PR description confirms it migrates both `PrivateKey` signing and related wallet flows.
- [#2383](https://github.com/multiversx/mx-sdk-rs/pull/2383) **Wallet signature refactor** — wraps raw `[u8; 64]` byte arrays and hex strings into a dedicated `WalletSignature` type with consistent serialization across SDK, snippets, and CLI. Filed and merged within ~3.5 hours of #2382 landing.

With these two merged, rc/v0.67 has: Bech32 HRP refactor, wallet refactor, ed25519-dalek signing, `WalletSignature` type — and CVE-2025-64170 (day 9) as the only known remaining merge. The RC is one dependabot approval from being tag-ready.

### 2. Rust 1.96 wasm build breakage — same-day detection

[#2386](https://github.com/multiversx/mx-sdk-rs/pull/2386) "wasm build fix for rust 1.96 + CI update" was opened at 23:39 UTC May 28, targeting **master** (not rc/v0.67). Rust 1.96 is a fresh stable toolchain release; the same-day opening suggests the breakage was caught in CI. The master target is significant: this is a base compatibility issue that will need to be either cherry-picked into rc/v0.67 or merged before the v0.67 tag. Watch for a backport PR.

### 3. Bootstrap interceptor reload — new feature cluster in mx-chain-go

A new branch `start-in-epoch-reload-interceptors` is accumulating work from multiple contributors:
- [#7862](https://github.com/multiversx/mx-chain-go/pull/7862) "more bootstrap optimizations" (raduchis, merged May 28) — additional correctness checks during bootstrap
- [#7861](https://github.com/multiversx/mx-chain-go/pull/7861) "recreate interceptors on bootstrap" (AdoAdoAdo, opened May 28)

This branch is distinct from the Supernova blocker cluster (#7841–#7857). The `start-in-epoch` prefix and the interceptor-reload focus suggest this targets a specific reliability case: a node that boots mid-epoch needs to reload block interceptors to process blocks it missed. This is operational robustness work, not a protocol-level Supernova feature. Two contributors in two days suggests a focused sprint.

3. **A bootstrap reliability track is running parallel to Supernova blockers.** The `start-in-epoch-reload-interceptors` cluster (two contributors, two PRs in one day) suggests the chain team is addressing a specific epoch-start scenario separately from the Supernova consensus changes. This may be a prerequisite for reliable Supernova testnet validator onboarding.
